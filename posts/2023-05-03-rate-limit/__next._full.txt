11:T143c,<p>수많은 클라이언트들의 동시다발적인 요청이 계속 보내지게 되면,
<br>서버에 과부하가 걸리고 결국에는 <code>shut down</code> 되는 위험이 있을 수 있습니다.</p>
<p>이러한 위험을 방지하기 위한 방법이 바로 요청에 제한을 두는 것입니다.</p>
<p>Rate Limiter 를 구현하는 방식으로는 다음과 같은 방법들이 있습니다.</p>
<h2>Rate Limiter 구현? 방법</h2>
<h3>API를 관리 플랫폼(<code>Business Products</code>) 사용하기</h3>
<ul>
<li>tyk.io, apigee, cloudflare 등으로 따로 구현하지 않고 사용만으로 처리할 수 있습니다.</li>
<li>단점으로는 비용이 비쌀 수 있다는 점과,</li>
<li>필요한 것 이상의 필요치 않는 기능을 위해 돈을 내야 된다는 점이 있습니다.</li>
</ul>
<h3>API를 관리 서비스(<code>Cloud Service</code>) 사용하기</h3>
<ul>
<li>AWS API Gateway, Google Cloud API Gateway</li>
<li>간편하긴 하지만 내 입맛에 맞게 조정하는 데는 한계가 있을 수 있습니다.</li>
</ul>
<blockquote>
<p>대부분 기업들은 자체적으로 Rate Limiter를 구현하는 경우가 많다고 합니다.</p>
</blockquote>
<h2>Rate Limit 필요성</h2>
<ul>
<li>Dos 공격(Denial-Of-Service attack)에 대한 시스템 보호할 수 있습니다.
<br>: 특정 API로 수많은 요청을 통해 서버를 다운 시키는 공격</li>
<li>365일 서버를 안정적으로 운영할 수 있습니다.</li>
<li>사용자들 간에 요청에 대한 공정성을 위해서 입니다.
<br>: 한 사용자의 너무 많은 요청으로, 다른 사용자가 이용하지 못하게 되지 않도록 방지</li>
<li>사용자가 유료/무료 버전에 따라 API를 제한할 수도 있습니다. (부가적인 기능)
<br>: Youtube API나 일반적인 외부 API의 경우에는 일일 횟수 제한이 있습니다.</li>
</ul>
<h2>Rate Limit Flow</h2>
<ul>
<li>A 클라이언트가 너무 많은 요청을 보내서 Rate Limit에 걸리는 경우,</li>
<li>서버는 <code>**429 Too Many Requests**</code> 상태코드 반환합니다.</li>
</ul>
<blockquote>
<p>429 응답을 반환할 때, 헤더에 [Retry-After: 2400] 정보를 함께 보내는 것을 권고하고 있습니다.
<br>(Retry-After: 얼마 후에 다시 요청해 달라는 요청)</p>
</blockquote>
<h2>Rate Limit 구현 방법</h2>
<ul>
<li>Global API Rate Limiting</li>
<li>Per User API Rate Limiting</li>
</ul>
<blockquote>
<p>일반적으로는 위의 2가지를 모두 함께 사용하게 됩니다.</p>
</blockquote>
<h3>🤔 Rate limiting in production?</h3>
<p>규모가 있는 경우에는 서버가 여러 인스턴스로, 여러 클러스터로 구성되는 경우가 많습니다.
이 경우에는 요청이 공평하게 분배 되도록 <strong>Load Balancer</strong>를 사용하게 됩니다.</p>
<p>그리고 Redis Store를 별도로 두어서 사용자 별로 요청 횟수를 기록합니다.
서버는 Redis 조회되는 정보로 요청의 승인/거부를 판단합니다.</p>
<h2>Rate Limit 알고리즘 종류</h2>
<h3>1. Fixed Window Counter</h3>
<ul>
<li>정해진 시간 단위로 window가 만들어 지고 요천 건수가 기록 됩니다.</li>
<li>window의 요청 건수가 정해진 건수보다 크면 해당 요청은 거부됩니다.</li>
</ul>
<blockquote>
<p>🔥 하지만 경계 시간대(18:59, 17:01에 몰리면)에 요청이 오면 두 배의 부하를 받게 됩니다.</p>
</blockquote>
<p>즉 구현은 쉽지만, 구간 경계의 트래픽 편향 문제가 발생됩니다.</p>
<h3>2. Sliding Window</h3>
<ul>
<li>Fixed Window Counter 단점의 경계 편향을 보안한 위한 알고리즘 입니다.</li>
<li>요청이 들어온 시간대 앞 뒤로 조금 조금씩 살펴 이전의 윈도우까지 살펴서 총 몇번의 요청이 들어왔는지 검사합니다.</li>
</ul>
<blockquote>
<p>🔥 하지만 window 요청건에 대해 로그 관리를 해야 하므로 구현과 메모리 비용이 높은 문제가 있습니다.</p>
</blockquote>
<h3>3. Leaky Bucket</h3>
<p>동시다발적으로 들어오는 요청들을 양동이에 담아놓고,
<br>양동이 밑에 구멍을 뚫어 천천히 떨어지도록 만드는 기법입니다.</p>
<img width=300 src="../../assets/img/docs/node/rate-limiting_leaky-bucket.png" />
<p>장점은 동시다발적으로 많은 요청이 들어와도 결국은 일정 간격으로 천천히 처리를 하기 때문에,
<br>서버의 과부화를 줄일 수 있습니다.</p>
<p>하지만 특정 시간대에 많은 요청이 필요한 경우라면,
<br>예를 들어 특정 시간대에 유명 가수의 콘서트 티켓을 예매해야 된다면,
<br>이 방식은 적절하지 않을 수 있습니다.</p>
<p>왜냐하면 양동에 물이 가득 차게되면 그 외에 요청들은 처리할 수 없기 때문입니다.</p>
